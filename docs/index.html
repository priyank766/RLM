Research Project · 2026

Recursive Language Models

A local, reproducible comparison of RLM vs vanilla LLM inference on long-context tasks — using the same model for both paths.

Priyank Patel
View Code Read the Paper Portfolio

Abstract

Standard LLMs suffer from context rot — a fundamental degradation in reasoning quality as input context grows, caused by attention dilution and the lost-in-the-middle effect. Recursive Language Models (RLMs), introduced by Zhang, Kraska, and Khattab (2025), take a different approach: instead of passing long documents into the model's context window, they store the document as a variable in an external Python REPL. The model then writes code to search, chunk, and query focused snippets — recursively calling itself on sub-problems. The full context never enters the token window.

This project implements an RLM system from scratch and runs a head-to-head comparison against vanilla inference using the same model (qwen3.5:2b) on both paths. All experiments run locally on a laptop GPU (RTX 4050, 6GB VRAM) via Ollama — no cloud, no API. The RLM answers questions about a 52,719-character document (beyond the vanilla model's visible context) in just 3 sub-calls, demonstrating that the approach works on commodity hardware with a 2B model.

1. Introduction

The context window arms race is one of the most visible trends in LLM development. Every few months, a new model announces a bigger window: 128K tokens, 256K, a million, ten million. It reads like Moore's Law for attention.

But there's a problem: bigger context windows don't actually solve the underlying issue. The phenomenon is called context rot — the measurable degradation in output quality as input context grows. It happens well before the model hits its maximum window. It happens on every model tested, from GPT-4.1 to Claude Opus 4 to Gemini 2.5 Pro.

The RLM paper offers a genuinely different approach. Instead of giving the model more context to read, it changes how the model interacts with context entirely. This project explores that idea by building the system from scratch.

2. The Problem: Context Rot

Context rot manifests in three well-understood mechanisms:

2.1 Attention Dilution

Transformer self-attention distributes a fixed budget across all tokens via softmax normalization. As context grows, the per-token attention weight shrinks — roughly proportional to 1/N. The relevant signal doesn't get louder; the noise floor rises. At 10,000 tokens, a model tracks ~100M pairwise relationships. At 100,000, that's 10 billion.

2.2 Lost-in-the-Middle Effect

Model attention follows a U-shaped curve across input positions. Information at the start (primacy) and end (recency) gets strong attention. Information in the middle drops by 30%+ in recall. This is baked into the architecture — not a training flaw.

2.3 Distractor Interference

Semantically similar but irrelevant tokens actively mislead the model. Coherent text makes this worse because plausible distractors look like the real thing. Code is especially vulnerable due to consistent naming conventions and repeated patterns.

Key insight: Standard fixes fail because they don't address the root cause. Bigger windows scale the problem. RAG can't capture multi-hop relationships. Summarization throws away the details you'll need later.

3. The RLM Solution

The Recursive Language Model architecture treats the long prompt as external state rather than text the model reads. Here's the core idea:

Vanilla LLM

[system + FULL 52,719-char document + query] → LLM → answer

Attention dilution degrades quality. KV cache truncation at 16K chars.

vs
RLM

[system + metadata: "52,719 chars"] → root LLM → writes Python code

↓

REPL executes code. Full document in context variable in RAM.

↓

keyword_windows("Holmes") → 3 sub-calls via llm_query()

↓

Root LLM consolidates → FINAL(answer)

Three key properties make this work:

  1. No passive context bloat — the model's working context stays small and focused
  2. Active exploration — the model writes code to find interesting sections, not just read passively
  3. Zero information loss — the full document sits in memory; nothing is summarized away

4. How It Works

4.1 The REPL Environment

A persistent Python environment initialized with the full document as a string variable:

namespace = {
    "context": <full 52,719-char document>,
    "query": "What was Holmes trying to figure out?",
    "context_len": 52719,
    # Helper functions:
    "head(n)", "tail(n)", "context_slice(start, end)",
    "chunk_text(size, overlap)", "keyword_windows(keyword, ...)",
    "regex_windows(pattern, ...)", "query_chunks(chunks, template)",
    # Recursive call:
    "llm_query(prompt)",
    # Termination:
    "FINAL(answer)", "FINAL_VAR(variable)",
}

4.2 The Root LLM Prompt

The model sees only metadata — never the full document:

System: You are an RLM. Write Python code to examine the context
variable and answer queries. Use helper functions like
keyword_windows(), llm_query(), and chunk_text().

User:
Context info:
  Total length : 52,719 characters
  Preview      : It is of the first importance to not allow yourself...

Query: What was Holmes trying to figure out?

4.3 The Loop

Each iteration: (1) root LLM generates code → (2) REPL executes it → (3) truncated output goes back to root LLM → (4) repeat until FINAL() or timeout (max 20 iterations).

4.4 Recursive Sub-Calls

When the root LLM calls llm_query(), a fresh invocation of the same model answers a focused question on a tiny snippet (~400–800 chars). No dilution. No distractors. Just a clean reading comprehension task.

4.5 Recursive-First Policy

For long contexts (≥16K chars), I enforce:

  • First-iteration FINAL() is always rejected — the model must explore first
  • FINAL() without sub-calls is rejected — evidence gathering required
  • Repeated code strategies are flagged and pushed toward variation

8. The REPL Engine

The REPL is built on Python's exec() with several safety features:

  • Persistent namespace — variables carry across iterations
  • Protected names — helper functions cannot be redefined by the model
  • Auto-restore — overwritten built-ins are restored after each execution
  • Code stripping — function definitions shadowing helpers are removed before exec
  • Error capture — both stdout and stderr are captured and returned
  • Final detection — FINAL() sets a flag checked after each execution

9. Recursive Sub-Calls

Each llm_query() call is a separate invocation of the same model with a focused prompt. The sub-LM has:

  • No memory of previous iterations
  • A clean, short context (~400–800 chars)
  • A direct, specific question
  • Instructions to answer concisely

This is the RLM advantage: instead of reading 50,000 words and hoping attention lands on the right parts, the model searches for interesting sections and reads only those. Three sub-calls for a 9,200-word document.

10. Training Pipeline

Each correct RLM trajectory is a training example. The pipeline:

  1. Collect trajectories — run benchmarks with --save-trajectories
  2. Filter — keep only correct answers with FINAL() and no REPL errors
  3. Format — convert to chat-style SFT data with build_sft_dataset.py
  4. Train — QLoRA SFT with AdamW or Muon via train_sft.py
  5. Evaluate — run the trained model through NIAH with eval_sft.py

The Muon optimizer ablation compares AdamW vs Muon (Newton-Schulz orthogonalization for 2D weight matrices) for RLM trajectory fine-tuning.
